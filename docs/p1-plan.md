# P1 기획 — 교통사고 사건 준비 클로즈드 베타

작성 기준일: 2026-07-21  
상태: P1 1차 기록. 현재 구현과 후속 의사결정은 [P1 v2 상세 기획·기술 명세](p1-v2-spec.md)가 우선한다.

## 1. 목표와 범위

P1은 P0의 공개 가이드·정적 목업을 사용자가 사건 준비와 상담 연결을 이해할 수 있는 클로즈드 베타 흐름으로 다듬는 단계다. 실제 인증, 저장, 파일 업로드, 법령·판례 연결, LLM 호출의 출시 여부는 각각 별도 보안·개인정보 검토를 거친다.

이번 1차 수정안의 구현 대상은 다음 네 가지다.

1. 고객용 `내 사건 공간`의 정보 구조와 질문 기반 예시 흐름을 구현한다.
2. 변호사 공개 프로필 정보 구조를 확장한다.
3. 일반 사용자 로그인 화면을 역할 선택형 레이아웃으로 재구성한다.
4. 개발자 전용 품질·운영 화면의 역할과 필수 지표를 정의한다.
5. AI 입력창은 제품 런타임 연결과 개발 보조 흐름을 분리해 설계한다.

### P1에서 하지 않는 일

- 과실비율, 승소 가능성, 합의금, 형량의 산정·예측·보장
- 실제 변호사 정보·사진·연락처 또는 실제 사건 데이터의 예시 사용
- 미검증 법령·판례 원문 및 개별 사건 대응 전략 제공
- 인증, 업로드, 저장, 결제, 상담 예약의 실제 운영 연결
- 브라우저가 Codex 터미널이나 개인 계정·자격증명에 직접 연결되는 구조

## 2. 고객용 내 사건 공간 1차 수정안

### 목표

보험사의 과실비율 제안에 이견이 있는 고객이 홈에서 곧바로 `내 사건 공간`에 들어가, 준비 상태·다음 행동·질문·자료 준비 항목을 한곳에서 확인하게 한다. 이 공간은 변호사 업무 포털과 분리된 고객용 작업 공간이며, 결과 예측이나 개별 법률자문을 제공하지 않는다.

### 진입과 메뉴

- 홈의 `내 사건`, `내 사건 공간 열기` 행동은 고객용 대시보드로 연결한다.
- 데스크톱은 좌측 메뉴, 모바일은 가로 스크롤 메뉴 또는 접힌 메뉴를 사용한다.
- 메뉴는 `내 사건 현황`, `나의 질문`, `자료 준비`, `질문 기반 예시`, `법전 참고`, `내 설정`으로 구성한다.

| 메뉴 | 고객 행동 | P1 1차 UI 제공 범위 |
|---|---|---|
| 내 사건 현황 | 준비 상태와 다음 행동 확인 | 사건 준비도, 확인한 자료 수, 다음 할 일 |
| 나의 질문 | 궁금한 점 메모 | 브라우저 내 질문 추가·수정, 질문별 다음 행동 안내 |
| 자료 준비 | 빠진 증거 확인 | 블랙박스, 현장 사진, 보험사 통화 기록, 진료·손해 자료, 상담 질문 체크 상태 |
| 질문 기반 예시 | 관련 쟁점의 예시 확인 | 로컬 핵심어 매칭, 예시 상황 요약, 확인 포인트 |
| 법전 참고 | 일반 확인 항목 확인 | 주제별 일반 확인 사항과 공식 원문·시행일 별도 확인 고지 |
| 내 설정 | 처리 원칙 확인 | 저장·공유·삭제·변호사 공유 동의가 실제 연결 전임을 안내 |

### 질문 기반 예시 원칙

- 입력한 질문에서 `교차로`, `과실`, `보험사`, `진단서`, `치료`, `증거` 등 사전에 정의한 핵심어만 찾아 로컬 예시를 필터링한다.
- 예시는 `관련 쟁점`, `짧은 상황 요약`, `확인할 증거·기록·절차`로만 구성한다. 판례 검색 결과나 실제 사건의 법률 판단으로 표현하지 않는다.
- 일치하는 예시가 없으면 무관한 결과를 보이지 않고, 사고 시점·진행 방향·보험사 안내·보유 자료를 더 구체화하도록 안내한다.
- 모든 예시에는 `일반적인 사건 준비 예시이며, 개별 사건의 법률 판단이나 결과 예측이 아닙니다.` 고지를 표시한다.

### 상태와 개인정보 경계

- P1 1차 화면 구현에서 질문 메모·체크 상태·예시 필터는 브라우저 메모리에서만 동작한다. 새로고침 뒤 유지·서버 저장·외부 전송은 하지 않는다.
- P1 클로즈드 베타에서 로그인 뒤 저장과 업로드를 검토할 때에는 동의, 접근권한, 보관기간, 삭제, 변호사 공유 동의를 별도 설계·검토한다.
- 원본 증거, 실제 이름·차량번호·사건번호, 실제 법령·판례 원문은 예시 데이터로 사용하지 않는다.

### 수용 기준

- 홈에서 고객용 내 사건 공간까지 끊김 없이 이동한다.
- 고객이 질문을 넣으면 관련 핵심어의 로컬 예시만 바뀌며, 질문이 저장·전송되지 않는다.
- 자료 준비와 다음 행동이 결과 예측 없이 표시된다.
- 320px 모바일 폭에서도 여섯 메뉴와 주요 행동을 이용할 수 있다.

## 3. 변호사 공개 프로필 1차 수정안

### 참고와 적용 제한

- `docs/references/lfind-lawyer-profile-shin-young-jun/README.md`의 프로필 히어로, 섹션 내비게이션, 데스크톱 상담 행동 영역과 모바일 세로 흐름을 정보 구조 참고로만 활용한다.
- `docs/references/index2/docs/superpowers/specs/2026-07-21-lawyer-portal-design.md`의 업무 포털 구조는 변호사 본인용 화면의 참고로만 활용한다.
- 두 자료의 사진, 로고, 실제 인물·사건·연락처·문구는 복제하지 않는다.

### 공개 프로필 정보 구조

1. **프로필 히어로**: 이니셜 기반 아바타(또는 동의를 받은 이미지), 이름, 소속, 검증 상태, 취급 분야, 활동 지역을 표시한다.
2. **고정 섹션 이동**: `소개`, `경력·자격`, `상담 안내`, `상담 전 준비`로 이동한다. 모바일은 가로 스크롤 탭, 데스크톱은 본문 상단 앵커를 사용한다.
3. **소개**: 변호사가 직접 확인한 소개와 취급 분야를 표시한다. 결과를 암시하는 문구는 허용하지 않는다.
4. **경력·자격**: 공개 동의·검증이 끝난 항목만 연도 순으로 표시한다. 수행 사건은 실명·사건번호·결과를 기본값으로 노출하지 않으며, 공개 검수된 익명화 전문 활동만 선택적으로 쓴다.
5. **상담 안내**: 상담 방식, 가능 지역, 가능 시간, 상담료(공개·검수된 경우), 예약 전 준비자료를 표시한다.
6. **행동 영역**: `상담 요청 준비하기`만 제공한다. 사건 분석 결과에 따른 추천, 광고성 우선 노출, 즉시 수임 유도는 제공하지 않는다.
7. **고지**: 광고·제휴 여부, 목록의 중립 정렬 기준, 실제 상담·결제 미연결 상태를 명확히 표시한다.

### 수용 기준

- 모바일 320px 폭에서 섹션 탭과 상담 행동이 가려지지 않는다.
- 모든 공개 정보는 예시 데이터 또는 검증·동의 상태를 명시한다.
- 프로필에서 개인 사건 분석, 결과 예측, 순위 근거를 사용하지 않는다.

## 4. 로그인 화면 1차 수정안

### 참고와 조정

`docs/references/login_example.html`의 좌·우 2단 레이아웃, 역할 탭, 소셜 로그인 배치를 참고한다. 다만 `24시간 AI 법률 상담`, `검증된 변호사 연결`처럼 서비스의 현재 제공 범위를 단정하는 문구는 사용하지 않는다.

### 화면 구성

- 좌측 안내: `사실을 정리하고, 다음 행동을 놓치지 않게`와 사건 저장·자료 정리·상담 준비라는 실제 제품 범위를 표시한다.
- 우측 폼: `의뢰자`와 `변호사` 역할 탭을 둔다.
  - 의뢰자: 사건 저장과 준비자료 관리 목적을 설명한다.
  - 변호사: 자격 검증 후 상담 요청과 공유 자료를 열람하는 목적을 설명한다.
- 이메일·비밀번호와 카카오·네이버·Google 버튼은 P1에서도 **연결 전 UI**임을 표시한다. 클릭 또는 제출은 실제 계정을 만들지 않고 해당 역할의 목업 화면으로만 이동한다.
- 모바일에서는 안내 영역 뒤에 폼을 세로로 배치하고, 키보드 포커스와 라벨을 유지한다.

### 수용 기준

- 역할 탭은 `role=tablist`/`role=tab`과 선택 상태를 제공한다.
- 폼 입력에는 표시 라벨과 자동완성 속성을 둔다.
- 실제 인증·소셜 제공자 연결 전에는 오해하지 않도록 목업 고지를 지속 노출한다.

## 5. 개발자 화면 기획

### 역할

개발자 화면은 일반 사용자·변호사·운영자 화면과 분리된 **비공개 스테이징 품질 화면**이다. 제품 사용자를 위한 대시보드나 접근 제어 수단이 아니며, 운영 배포본에서는 서버 측 관리자 권한과 감사 로그가 구현되기 전 공개하지 않는다.

### 필수 지표

P1에서는 실제 개인 사건 내용 없이 집계·익명 예시 데이터만 표시한다.

| 영역 | 지표 | 필요한 이유 |
|---|---|---|
| 퍼널 | 무료 가이드 시작·완료율, 로그인 유도 클릭률, 사건 저장 시도율 | P0 핵심 흐름의 이탈 지점 확인 |
| 준비 | 체크리스트 항목 완료율, 다음 행동 완료율, 자료 누락 항목 수 | 결과 예측이 아닌 사건 준비도 품질 확인 |
| AI 품질 | 구조화 응답 성공률, 사용자 수정률, 추가 확인 항목 비율, 안전 문구 누락률 | 사실·추정 구분과 안전 응답 검증 |
| 출처·안전 | 출처 연결률, 법령 기준일 누락률, 금지 판단 표현 감지 건수 | 법률정보 품질과 안전 경계 점검 |
| 운영 | 변호사 검증 대기, 상담 요청 응답시간, 삭제 요청 처리시간 | 베타 운영 병목 확인 |
| 기술 | API 오류율, 응답시간, 호출량, 한도 초과·봇 차단 건수 | 비용·남용·장애 대응 |

### 화면 구성과 권한

- 요약 카드, 기간 필터, 익명화된 품질 샘플, 경고 목록, 지표 정의·산식·기준일을 제공한다.
- 원본 증거, 사용자 이름·연락처·사건번호, 프롬프트 전문, 비식별화 전 AI 입력은 표시하지 않는다.
- P1 목업 진입점은 기존처럼 일반 메뉴에 노출하지 않는다. 운영 전환 시에는 서버 측 역할 기반 권한, 감사 로그, 접근 만료를 요구한다.

## 6. LLM 입력창과 Codex 터미널 판단

### 결론: 개발자의 수동 텍스트 검토·답변 생성은 가능

개발자는 Codex 터미널에 글 형태의 입력을 붙여 넣고, 정해 둔 출력 형식의 답변을 받을 수 있다. 이는 개발·기획 검토 흐름으로는 적합하며, P1에서 API 연결 없이 AI 응답 형태와 안전 문구를 점검하는 방법으로 사용한다.

다만 Codex 터미널은 개발자가 로컬 작업을 수행하는 도구이므로, 배포된 브라우저가 직접 호출하는 제품용 백엔드나 사용자 요청을 지속 처리하는 런타임으로 사용하지 않는다. 배포 웹에서 개발자 PC의 터미널을 호출하거나 인터넷에 노출하는 방식은 인증·권한·개인정보·감사·가용성 경계를 충족하지 못한다.

OpenAI도 브라우저에 비밀 키를 두지 말고, 요청을 비밀값을 보관하는 자체 백엔드로 보내도록 안내한다. [OpenAI API 키 안전 수칙](https://help.openai.com/en/articles/5112595-best-practices-for-api-key-safety)

### P1의 수동 검토 흐름

1. **입력 준비**: 개발자는 실제 개인정보를 제거한 샘플 또는 동의·처리 정책이 확인된 최소 텍스트만 준비한다.
2. **Codex 질의**: `사용자 직접 진술`, `추가 확인사항`, `변호사에게 물어볼 질문`, `안전 고지`의 JSON 형식만 반환하도록 프롬프트한다. 과실·승소·합의금·형량 판단은 금지한다.
3. **사람 검토**: 개발자가 출력의 사실·추정 구분, 금지 표현, 개인정보 재노출 여부를 확인한다.
4. **반영 범위**: 검토를 통과한 내용은 테스트 fixture·화면 목업·기획 문서에 수동 반영한다. Codex 출력이 사용자에게 자동 전달되거나 제품 데이터로 자동 저장되지는 않는다.
5. **제품 UI**: 실제 API 연결 전 입력창은 `개발 검토용 시연` 또는 `연결 준비 중` 상태를 명시하며, 사용자가 입력한 내용을 전송하지 않는다.
6. **실제 연결 시점(P1 이후 승인 필요)**: Cloudflare Worker의 `POST /api/ai-draft`가 서버 환경변수의 프로젝트 전용 API 키로 LLM API를 호출한다. 로그인, 권한 확인, 호출 한도, 봇 차단, 구조화 출력 검증, 감사·삭제 정책을 갖춘 뒤 제한된 베타에서만 활성화한다.

### 의도 기반 오용 방지 배리어

P1의 수동 Codex 검토에서도 입력 의도에 따라 답변 범위를 제한할 수 있다. 다만 이는 개발자 검토를 돕는 절차이지, 사용자를 자동 차단하는 보안 통제가 아니다.

| 감지할 의도·신호 | P1 수동 검토 조치 | 실제 API 연결 후 서버 조치 |
|---|---|---|
| 과실비율·승소·합의금·형량의 단정 또는 보장 요구 | 결과를 제시하지 않고 자료 확인·변호사 검토 안내로 전환 | 요청 전·후 정책 검사와 안전한 대체 응답 |
| 상대방을 속이거나 증거를 조작·은닉하려는 요구 | 도움을 거절하고 적법한 증거 보존·전문가 상담 안내 | 차단, 최소 감사 이벤트 기록, 반복 시 제한 |
| 프롬프트·내부 지침·비밀값을 추출하려는 요구 | 지침·비밀값을 공개하지 않고 작업 목적의 범위로 복귀 | 시스템 지침 분리, 비밀값 비노출, 입력·출력 필터 |
| 이름·연락처·차량번호·사건번호 등 과도한 개인정보 | 가림·삭제를 요청하고 원문을 fixture에 반영하지 않음 | 입력 최소화·마스킹·보관 제한·권한 검사 |
| 반복·대량 요청 또는 서비스 외 목적의 사용 | 검토를 중단하고 목적을 확인 | 로그인·호출 한도·봇 차단·속도 제한 |

모든 배리어는 의도를 완벽하게 판별한다고 가정하지 않는다. 차단 사유는 짧고 중립적으로 알리고, 허용되는 범위의 대안(사실 정리, 증거 보존, 일반 절차 정보, 변호사 상담 준비)을 제시한다. 실제 API 연결 전에는 클라이언트 측 키워드 경고만으로 민감정보 보호나 접근 통제를 주장하지 않는다.

### 사용자 노티 문구

- 연결 전: `AI 초안 기능은 현재 준비 중입니다. 입력한 내용은 이 화면에서 저장·전송되지 않습니다.`
- 실제 연결 후: `AI가 입력한 사실을 정리한 참고용 초안입니다. 법률 판단이나 대응 전략이 아니며, 불확실한 내용은 추가 확인 또는 변호사 검토가 필요합니다.`

## 7. Luna 구현 순서

1. 현재 P0 화면과 라우트에 대한 회귀 확인 및 P1 화면별 수용 기준을 테스트로 작성한다.
2. 고객용 내 사건 공간의 메뉴·질문 메모·자료 준비·핵심어 기반 예시를 브라우저 상태만으로 구현한다.
3. 로그인 재구성과 역할별 목업 이동을 구현한다.
4. 변호사 목록·프로필을 위 정보 구조와 반응형 섹션 내비게이션으로 개편한다.
5. 비공개 개발자 품질 화면을 익명 집계 목업으로 구현한다.
6. AI 입력창은 연결 대기 UI와 명확한 노티만 구현한다. 개발자는 Codex 터미널에서 비식별화 입력을 수동 검토할 수 있으나, Codex 또는 실제 LLM 자동 호출은 추가 승인 전 구현하지 않는다.
7. 키보드 접근성, 모바일 320px·768px·데스크톱 확인, `npm run build`를 검증한다.

## 8. 완료 정의

- 이 문서의 다섯 영역이 P1 화면과 테스트에 반영된다.
- 모든 목업·연결 대기 상태가 실제 기능으로 오인되지 않는다.
- 의도 기반 오용 방지 기준은 수동 검토 프롬프트와 체크리스트에 반영되고, 실제 자동 차단은 서버 측 정책으로만 구현한다.
- 법률 안전, 개인정보 최소화, 변호사 중립 정렬 원칙을 위반하지 않는다.
- 실제 LLM 연결은 별도 보안·운영 승인 없이는 활성화되지 않는다.
