# 제품 결정 기록

## 확정된 결정

### 2026-07-21

- 첫 제품 범위는 종합 법률상담이 아니라 교통사고 사건 준비로 제한한다.
- 초기 핵심 상황은 보험사 과실비율 이견과 분심위 이후 단계다.
- 프론트엔드 검증부터 시작한다.
- 무료 가이드와 템플릿은 비로그인으로 제공한다.
- 사건 저장, 자료 업로드, AI 정리는 로그인 후 제공한다.
- AI는 사실 정리와 출처 기반 법률정보를 담당하고 구체적 법률자문은 변호사가 담당한다.
- 개별 사건 분석을 변호사 순위에 사용하지 않는다.
- 일반 사용자, 변호사, 운영 관리자 화면을 별도로 둔다.
- 승소 가능성, 형량, 합의금, 과실비율을 확정하거나 보장하지 않는다.
- 첫 완결 흐름은 `상황 선택 → 무료 체크리스트 → 로그인 유도`다.

### 2026-07-21 — P0 프론트엔드 기술 기준

- P0 웹 앱은 `React + TypeScript + Vite`로 구현한다. 서버 기능이 없는 화면 검증 단계에서 빠른 개발·정적 배포·낮은 운영 복잡도를 우선한다.
- 라우팅은 `React Router`의 클라이언트 라우팅을 사용한다. P0에서는 공개 화면과 정적 목업만 두며, 인증·파일·AI 호출은 구현하지 않는다.
- 스타일은 `Tailwind CSS`와 프로젝트 내부의 재사용 UI 컴포넌트 조합으로 구성한다. 접근성 속성과 상태(기본·포커스·오류·비활성)를 컴포넌트 기준으로 통일한다.
- P0는 `Cloudflare Pages` 정적 배포를 기본 대상으로 한다. 배포 환경 변수나 비밀값은 사용하지 않으며, 향후 개인정보·AI 기능은 별도 서버 API를 통해서만 연결한다.
- 화면 범위는 P0-A와 P0-B로 구분한다. P0-A는 `상황 선택 → 과실비율 이견 입력 → 무료 체크리스트 → 로그인 유도`의 완결 흐름이고, P0-B는 사건·AI 리포트·변호사·관리자 화면의 정적 목업이다.

### 2026-07-21 — P0 UI 구현 결정

- 첫 화면은 `사실을 정리하고, 다음 행동을 놓치지 않게`라는 메시지와 상황 선택 메뉴를 함께 보여준다. 사용자가 법률 용어를 몰라도 시작할 수 있도록 `보험사 과실비율이 억울해요`를 대표 진입점으로 둔다.
- 과실비율 이견 가이드는 사고 시기, 보험사와의 현재 대화 상태, 자유 메모의 3단계로 구성한다. 비로그인 입력은 체크리스트를 만드는 동안만 사용하고 서버 저장은 하지 않는다.
- 체크리스트는 과실비율·승소 가능성 같은 결과 예측 대신 블랙박스 원본, 현장 사진, 보험사 통화 기록, 상담 질문의 준비 상태를 보여준다.
- 사건 준비도는 자료 완성도를 나타내는 목업 수치로만 노출하고, 법률적 판단 또는 변호사 목록 정렬에는 사용하지 않는다.
- AI 사건 리포트 목업은 `사용자가 직접 입력한 사실`, `자료에서 확인된 내용`, `아직 확인이 필요한 내용`, `변호사에게 물어볼 질문`을 별도 카드로 구분한다. P0에는 실제 AI·법령·판례 출처 연결이 없다.
- 변호사 목록은 지역·취급분야 등 객관적 필터와 `중립 순환 정렬·광고 없음` 안내를 사용한다. 사건 내용에 기반한 추천 순위와 결과 예측은 제공하지 않는다.

### 2026-07-21 — 변호사 프로필 화면 참고자료 관리

- 엘파인드의 신영준 변호사 공개 프로필은 변호사 소개 상세 화면의 정보 구조와 반응형 화면 구성 참고자료로 `docs/references/lfind-lawyer-profile-shin-young-jun/`에 기록한다.
- 변호사 소개·목록 화면을 구현하거나 수정할 때 해당 참고자료를 먼저 확인한다. 다만 원문 디자인·문구·사진·연락처를 복제하지 않으며, 서비스의 중립 정렬·광고 고지·개인정보 최소화 원칙을 우선한다.

### 2026-07-21 — 로그인 역할 구분 및 개발자 목업 진입점

- 루트 `index.html`의 로그인 화면은 `의뢰자`와 `변호사`를 명확히 구분한다. 의뢰자는 사건 정리·자료 보관·상담 요청, 변호사는 자격 검증 후 상담 요청과 공유 자료를 열람하는 역할로 안내한다.
- P0 테스트를 위한 개발자 UI는 일반 로그인 화면에 링크를 표시하지 않고 `index.html#dev-console`로만 연다. 이는 보안 기능이 아닌 목업 진입점이며, 실제 운영에서는 서버 측 역할 기반 권한 검증과 감사 로그를 적용해야 한다.

### 2026-07-21 — LLM 서버 호출 및 Cloudflare 배포 준비

- LLM API 키는 프론트엔드에 두지 않고 Cloudflare Pages Functions의 `POST /api/ai-draft`에서만 사용한다. Pages의 Secret으로 `LLM_API_KEY`를, 일반 환경 변수로 `LLM_API_URL`과 `LLM_MODEL`을 설정한다.
- 이 API는 OpenAI 호환 Chat Completions 형식으로 요청하고, 사실·추가 확인사항·변호사 질문으로 제한된 JSON만 반환한다. 과실비율, 승소 가능성, 합의금, 형량의 판단·예측·보장을 금지한다.
- Cloudflare Pages 배포는 `apps/web`을 루트로 하여 `npm run build`와 `dist`를 사용한다. `functions/` 디렉터리는 Pages Functions로 함께 배포한다.
- 인증 전 공개 API는 비용·남용 위험이 있으므로 현재 엔드포인트는 기능 연결용 최소 구현으로만 둔다. 실제 사용자 데이터 처리 전에는 인증, 호출 한도, 봇 차단, 감사·삭제 정책을 추가한다.
