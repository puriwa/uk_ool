# 제품 결정 기록

## 확정된 결정

### 2026-07-21

- 첫 제품 범위는 종합 법률상담이 아니라 교통사고 사건 준비로 제한한다.
- 초기 핵심 상황은 보험사 과실비율 이견과 분심위 이후 단계다.
- 프론트엔드 검증부터 시작한다.
- 무료 가이드와 템플릿은 비로그인으로 제공한다.
- 사건 저장, 자료 업로드, AI 정리는 로그인 후 제공한다.
- AI는 사실 정리와 출처 기반 법률정보를 담당하고 구체적 법률자문은 변호사가 담당한다.
- 개별 사건 분석을 변호사 순위에 사용하지 않는다.
- 일반 사용자, 변호사, 운영 관리자 화면을 별도로 둔다.
- 승소 가능성, 형량, 합의금, 과실비율을 확정하거나 보장하지 않는다.
- 첫 완결 흐름은 `상황 선택 → 무료 체크리스트 → 로그인 유도`다.

### 2026-07-21 — P0 프론트엔드 기술 기준

- P0 웹 앱은 `React + TypeScript + Vite`로 구현한다. 서버 기능이 없는 화면 검증 단계에서 빠른 개발·정적 배포·낮은 운영 복잡도를 우선한다.
- 라우팅은 `React Router`의 클라이언트 라우팅을 사용한다. P0에서는 공개 화면과 정적 목업만 두며, 인증·파일·AI 호출은 구현하지 않는다.
- 스타일은 `Tailwind CSS`와 프로젝트 내부의 재사용 UI 컴포넌트 조합으로 구성한다. 접근성 속성과 상태(기본·포커스·오류·비활성)를 컴포넌트 기준으로 통일한다.
- P0는 `Cloudflare Pages` 정적 배포를 기본 대상으로 한다. 배포 환경 변수나 비밀값은 사용하지 않으며, 향후 개인정보·AI 기능은 별도 서버 API를 통해서만 연결한다.
- 화면 범위는 P0-A와 P0-B로 구분한다. P0-A는 `상황 선택 → 과실비율 이견 입력 → 무료 체크리스트 → 로그인 유도`의 완결 흐름이고, P0-B는 사건·AI 리포트·변호사·관리자 화면의 정적 목업이다.

### 2026-07-21 — P0 UI 구현 결정

- 첫 화면은 `사실을 정리하고, 다음 행동을 놓치지 않게`라는 메시지와 상황 선택 메뉴를 함께 보여준다. 사용자가 법률 용어를 몰라도 시작할 수 있도록 `보험사 과실비율이 억울해요`를 대표 진입점으로 둔다.
- 과실비율 이견 가이드는 사고 시기, 보험사와의 현재 대화 상태, 자유 메모의 3단계로 구성한다. 비로그인 입력은 체크리스트를 만드는 동안만 사용하고 서버 저장은 하지 않는다.
- 체크리스트는 과실비율·승소 가능성 같은 결과 예측 대신 블랙박스 원본, 현장 사진, 보험사 통화 기록, 상담 질문의 준비 상태를 보여준다.
- 사건 준비도는 자료 완성도를 나타내는 목업 수치로만 노출하고, 법률적 판단 또는 변호사 목록 정렬에는 사용하지 않는다.
- AI 사건 리포트 목업은 `사용자가 직접 입력한 사실`, `자료에서 확인된 내용`, `아직 확인이 필요한 내용`, `변호사에게 물어볼 질문`을 별도 카드로 구분한다. P0에는 실제 AI·법령·판례 출처 연결이 없다.
- 변호사 목록은 지역·취급분야 등 객관적 필터와 `중립 순환 정렬·광고 없음` 안내를 사용한다. 사건 내용에 기반한 추천 순위와 결과 예측은 제공하지 않는다.

### 2026-07-21 — 변호사 프로필 화면 참고자료 관리

- 엘파인드의 신영준 변호사 공개 프로필은 변호사 소개 상세 화면의 정보 구조와 반응형 화면 구성 참고자료로 `docs/references/lfind-lawyer-profile-shin-young-jun/`에 기록한다.
- 변호사 소개·목록 화면을 구현하거나 수정할 때 해당 참고자료를 먼저 확인한다. 다만 원문 디자인·문구·사진·연락처를 복제하지 않으며, 서비스의 중립 정렬·광고 고지·개인정보 최소화 원칙을 우선한다.

### 2026-07-21 — 로그인 역할 구분 및 개발자 목업 진입점

- 루트 `index.html`의 로그인 화면은 `의뢰자`와 `변호사`를 명확히 구분한다. 의뢰자는 사건 정리·자료 보관·상담 요청, 변호사는 자격 검증 후 상담 요청과 공유 자료를 열람하는 역할로 안내한다.
- P0 테스트를 위한 개발자 UI는 일반 로그인 화면에 링크를 표시하지 않고 `index.html#dev-console`로만 연다. 이는 보안 기능이 아닌 목업 진입점이며, 실제 운영에서는 서버 측 역할 기반 권한 검증과 감사 로그를 적용해야 한다.

### 2026-07-21 — LLM 서버 호출 및 Cloudflare Workers 배포 준비

- Cloudflare의 현재 Git 연결 화면은 Worker Builds 기반이므로, Vite 정적 자산과 LLM API를 `apps/web/src/worker.ts` 및 `wrangler.jsonc`의 Worker 정적 자산 구성으로 배포한다. Worker 이름은 Cloudflare 프로젝트명과 같은 `ukool`로 둔다.
- LLM API 키는 프론트엔드에 두지 않고 Worker의 `POST /api/ai-draft`에서만 사용한다. Worker Secret으로 `LLM_API_KEY`를, 일반 환경 변수로 `LLM_API_URL`과 `LLM_MODEL`을 설정한다.
- 이 API는 OpenAI 호환 Chat Completions 형식으로 요청하고, 사실·추가 확인사항·변호사 질문으로 제한된 JSON만 반환한다. 과실비율, 승소 가능성, 합의금, 형량의 판단·예측·보장을 금지한다.
- Cloudflare Workers Git 배포는 `apps/web`을 루트로 하여 `npm run build` 후 `npx wrangler deploy`를 실행한다. `wrangler.jsonc`가 `dist` 정적 자산과 API 경로를 함께 배포한다.
- 인증 전 공개 API는 비용·남용 위험이 있으므로 현재 엔드포인트는 기능 연결용 최소 구현으로만 둔다. 실제 사용자 데이터 처리 전에는 인증, 호출 한도, 봇 차단, 감사·삭제 정책을 추가한다.

### 2026-07-21 — P1 1차 수정안 기획 기준

- P1 구현 범위와 수용 기준은 `docs/p1-plan.md`에서 관리한다. 변호사 공개 프로필, 역할 선택형 로그인, 비공개 개발자 품질 화면, AI 입력창의 연결 대기 상태를 1차 수정안으로 둔다.
- 고객용 내 사건 공간은 P1 1차 수정안에 포함한다. `내 사건 현황`, `나의 질문`, `자료 준비`, `질문 기반 예시`, `법전 참고`, `내 설정`을 제공하며, 질문·체크 상태·예시 필터는 실제 저장·전송 없이 브라우저 상태로만 동작한다.
- 질문 기반 예시는 실제 판례 검색이나 개별 사건 적용이 아니라 사전에 정의한 핵심어를 이용한 로컬 예시 필터다. 일치 항목이 없으면 추가 사실·증거·절차 확인을 안내하고, 모든 예시에 법률 판단·결과 예측이 아니라는 고지를 표시한다.
- 변호사 프로필은 `docs/references/lfind-lawyer-profile-shin-young-jun/`과 `docs/references/index2/`의 정보 구조만 참고한다. 실제 인물·사건·사진·문구를 복제하지 않고, 검증 가능한 공개 정보와 중립 정렬·광고 고지를 우선한다.
- 로그인 화면은 `docs/references/login_example.html`의 레이아웃을 참고하되, 실제 인증·소셜 로그인 연동 전에는 목업 상태를 명확히 고지한다.
- 개발자 화면은 일반 사용자에게 노출하지 않는 스테이징 품질 화면으로 한정한다. 실제 개인 사건 데이터 대신 익명 집계·예시 데이터만 표시하며, 운영 전환에는 서버 측 역할 권한과 감사 로그가 필요하다.
- 개발자는 P1에서 비식별화된 글 입력을 Codex 터미널에 수동으로 넣어 구조화된 답변 초안을 받을 수 있다. 이 출력은 사람이 검토한 뒤 테스트 fixture·목업·기획에만 수동 반영하며, 사용자에게 자동 전달하거나 제품 데이터로 자동 저장하지 않는다.
- P1 수동 검토 프롬프트와 체크리스트에는 결과 보장 요구, 증거 조작·은닉, 프롬프트·비밀값 추출, 과도한 개인정보, 반복·대량 요청의 의도 기반 오용 방지 기준을 둔다. 이는 개발 검토용 배리어이며, 실제 사용자 자동 차단은 서버 측 정책 검사·권한·호출 한도·감사 체계가 갖춰진 뒤에만 구현한다.
- Codex 터미널은 배포 웹앱의 백엔드나 사용자 요청 처리 경로로 사용하지 않는다. 실제 LLM 호출은 별도 승인 후 서버 측 Worker와 비밀값·권한·한도·감사 정책을 갖춰 연결한다.

### 2026-07-21 — P1 v2 사용자 흐름 재설계

- 홈의 `무료로 내 상황 확인하기`는 상담 방식 선택 화면으로, `3분 가이드 시작`은 설명형 가이드로 분리한다.
- 비로그인 기능은 `3분 가이드`와 규칙 기반 `간단 확인`으로 구성한다. 3분 가이드는 상황별 핵심 설명 3개 뒤에 체크리스트를 제공한다.
- 로그인 후 `무료 AI 사건정리`는 분야별 선택, 자유 서술, 보험사 의견, 파일 첨부, 구조화 결과의 흐름으로 구성한다.
- 상담 분야는 사고 직후, 과실비율 이견, 접수 거부, 분심위 결과 이견, 경찰 조사, 합의 검토의 여섯 분기로 시작한다.
- AI 질문 전에 `답변 주체: AI (Codex)`와 법률자문이 아니라는 고지를 표시하고 사용자가 확인해야 실행할 수 있게 한다.
- `내 사건`과 AI 사건정리는 로그인 뒤에 둔다. P1 사건 결과는 브라우저 저장 목업이며 실제 서버 사건 저장으로 표현하지 않는다.

### 2026-07-21 — 로컬 Codex CLI 제품 흐름 검증

- 동작 우선의 P1 검증을 위해 Vite 개발 서버에 로컬 전용 `POST /api/codex-draft`를 둔다.
- 개발 서버는 저장된 Codex CLI 인증으로 `codex exec --ephemeral --sandbox read-only --output-schema`를 실행한다.
- 기본 모델은 명확하고 반복적인 구조화 작업에 맞는 `gpt-5.6-luna`로 두고 `UKOOL_CODEX_MODEL`로 재정의할 수 있게 한다.
- 사진은 최대 3개를 실제 Codex 이미지 입력으로 전달한다. 영상과 PDF는 P1에서 파일 정보만 전달하며 분석했다고 표현하지 않는다.
- 로컬 Codex 브리지는 Vite 개발 모드에서만 활성화하고 Cloudflare 운영 Worker에는 포함하지 않는다.
- 제품 운영 LLM 경로는 별도 JWT 검증, 한도, 감사, 데이터 정책을 갖춘 뒤 다시 결정한다.

### 2026-07-21 — 관리형 로그인 기본안

- 자체 비밀번호 DB를 운영하지 않고 Supabase Auth를 P1의 관리형 인증 기본안으로 선택한다.
- `VITE_SUPABASE_URL`과 `VITE_SUPABASE_ANON_KEY`가 있으면 Google·Kakao OAuth를 사용하고, 없으면 전체 흐름 검증을 위한 더미 로그인을 사용한다.
- Naver는 Supabase 기본 소셜 제공자에 없으므로 P1에서는 더미로 유지하고 별도 OAuth/OIDC 연동을 검토한다.
- 소셜 제공자마다 배경색과 이모티콘이 있는 계정 배지를 제공한다.
- 실제 사건 API는 프론트엔드 로그인 상태가 아니라 서버가 검증한 JWT와 역할을 기준으로 권한을 판단해야 한다.

### 2026-07-21 — 공개 변호사 정보 수집 표시

- 엘파인드의 `교통사고` 검색 결과와 몇 개 상세 프로필에서 공개된 이름, 소속, 수행분야, 이력, 학력, 수행사건 제목·법원·연도를 수집해 P1 정보 구조를 검증한다.
- 각 프로필에 원문 URL, 확인일, `엘파인드 공개 정보에서 수집` 고지를 표시한다.
- 엘파인드 또는 해당 변호사와의 제휴·추천으로 표현하지 않고, 사건 분석 결과를 노출 순위에 사용하지 않는다.
- 사진, 전화번호, 이메일, 상세 주소, 장문의 홍보 문구는 복제하지 않는다.
- 실제 운영 공개와 자동 정기 수집 전에는 당사자 확인, 최신성 검수, 이용약관·robots·재사용 범위를 확인한다.
