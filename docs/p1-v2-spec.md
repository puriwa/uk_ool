작성 기준일: 2026-07-21  
상태: 구현·기능 검증 반영안

## 1. 재기획 결론

P1의 첫 사용자 경험을 다음 세 흐름으로 명확히 분리한다.

1. `3분 가이드`: 비로그인, AI 미사용, 설명 중심
2. `간단 확인`: 비로그인, 선택형 규칙 결과, 파일·자유글 미사용
3. `무료 AI 사건정리`: 로그인, 자유글·보험사 의견·자료 첨부, Codex 구조화 결과

홈의 `무료로 내 상황 확인하기`는 `/consult`, `3분 가이드 시작`은 `/guide/insurance-dispute`로 연결한다. 두 버튼은 더 이상 같은 화면으로 이동하지 않는다.

`내 사건`과 `무료 AI 사건정리`는 로그인 뒤에 둔다. P1에서는 관리형 Auth가 설정되지 않아도 전체 흐름을 시험할 수 있도록 더미 로그인 폴백을 유지한다.

## 2. P1 사용자 가치

사용자는 로그인 전에도 다음 만족을 얻어야 한다.

- 법률 용어를 몰라도 자기 상황을 고를 수 있다.
- 상황에 따라 다른 설명 세 개와 체크리스트를 받는다.
- 간단 확인에서 지금 단계에 맞는 준비 항목을 즉시 본다.
- AI를 원하지 않는 사용자는 여기서 완료할 수 있다.

로그인한 사용자는 추가로 다음 만족을 얻는다.

- 사건 설명과 보험사 의견을 한 번에 입력한다.
- 사진, 영상, PDF를 첨부 목록으로 관리한다.
- AI가 사용자 진술, 자료 사실 후보, 확인 필요, 다음 행동을 분리한다.
- 결과를 `내 사건`에서 다시 보고 변호사 프로필로 이어간다.

## 4. 상담 분기

P1 교통사고 범위에서 다음 여섯 영역을 제공한다.

| 영역 | 후속 질문 예시 | 가이드 초점 |
|---|---|---|
| 사고 직후 | 현장·당일·부상 여부 | 안전, 신고, 원본 보존 |
| 보험사 과실비율 이견 | 비율만 안내·근거 요청·이의 전달 | 숫자와 근거 분리 |
| 대인·대물 접수 거부 | 대인·대물·둘 다·불명 | 거부 주체와 기록 |
| 분심위 결과 이견 | 결과만·결정문·보험사 설명 | 결정 이유와 제출자료 |
| 경찰 조사 | 전화·출석·자료 제출 | 기억과 자료 구분 |
| 합의 검토 | 전화 금액·문서·치료 중 | 금액과 조건 분리 |

영역을 바꾸면 후속 선택지, 3분 가이드, 체크리스트가 함께 바뀌어야 한다.

## 5. 간단 확인과 무료 AI 사건정리의 차이

| 항목 | 간단 확인 | 무료 AI 사건정리 |
|---|---|---|
| 목표 | 지금 확인할 일반 항목 찾기 | 내 사건 자료를 상담 가능한 형태로 정리 |
| 로그인 | 없음 | 필요 |
| 입력 | 영역·진행 단계 | 영역·진행 단계·자유글·보험사 의견·파일 |
| 처리 | 로컬 규칙 | 로컬 Codex CLI |
| 출력 | 고정 체크리스트 | 사건별 구조화 JSON |
| 저장 | 없음 | P1은 브라우저 `localStorage` 목업 |
| 법률 판단 | 없음 | 없음 |

사용자에게 `무료 상담`이라는 표현을 쓰더라도, 화면 첫 줄에서 `답변 주체: AI (Codex)`와 법률자문이 아님을 고지한다.

## 6. AI 입력·출력 명세

### 입력

- `topic`: 상담 영역
- `answers`: 분야별 선택 답변
- `incidentText`: 사건 서술, 최대 8,000자
- `insurerPosition`: 보험사 의견, 최대 3,000자
- `attachments`: 최대 8개 파일의 이름·MIME·크기
- 이미지: 최대 3개, 개별 4MB 이하를 Codex 이미지 입력으로 전달
- 영상·PDF: P1에서는 파일 목록만 전달

영상은 사용자가 중요 시점 캡처를 사진으로 추가하거나 핵심 내용을 글로 적도록 안내한다. 영상 전체를 분석했다고 표시하지 않는다.

### 출력 JSON

```text
userFacts[]
evidenceFacts[{ statement, source }]
uncertainties[]
generalInformation[]
nextActions[{ title, reason, urgency }]
questionsForLawyer[]
notice
```

필수 화면 구분:

- 사용자가 직접 말한 사실
- 사진에서 확인된 사실 후보
- 아직 확인이 필요한 내용
- 일반적으로 알아둘 정보
- 다음 행동
- 변호사에게 물어볼 질문
- AI 고지

### 금지 출력

- 과실비율 산정 또는 확정
- 승소 가능성·형량·합의금 예측
- 근거 없는 법령·판례 단정
- 사진에서 보이지 않는 번호판·얼굴·행동 추측
- 영상·PDF 내용을 실제 확인했다는 표현

## 7. 로컬 Codex 실행

P1 기능 검증에서는 배포된 Worker가 아니라 Vite 개발 서버의 `/api/codex-draft`가 로컬 `codex exec`를 실행한다.

```text
브라우저
→ Vite 개발 미들웨어
→ 임시 이미지 파일 생성
→ codex exec --ephemeral --sandbox read-only
→ --model gpt-5.6-luna
→ --output-schema codex-output.schema.json
→ JSON 검증·화면 표시
```

구현 특성:

- 저장된 Codex CLI 로그인을 재사용한다.
- 프롬프트는 표준입력으로 전달해 명령 인자 삽입을 피한다.
- 120초 후 실행을 중단한다.
- 임시 이미지는 응답 뒤 삭제한다.
- 프로덕션 Vite 빌드와 Cloudflare Worker에는 이 미들웨어가 포함되지 않는다.
- `UKOOL_CODEX_MODEL`로 모델을 바꿀 수 있으며 기본값은 `gpt-5.6-luna`다.

실행:

```bash
cd apps/web
npm run dev
```

Codex CLI 로그인이 되어 있어야 한다. 2026-07-21 기능 검증에서 비식별 교차로 예시가 약 27초에 구조화 결과를 반환했다.

## 8. 로그인

### 결론

자체 비밀번호 DB를 운영하지 않고 `Supabase Auth`를 기본 관리형 인증으로 사용한다. P1 개발 중에는 설정 유무에 따라 다음처럼 동작한다.

- `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` 있음: Google·Kakao OAuth
- 설정 없음: 색상·이모티콘이 있는 더미 로그인
- Naver: P1에서는 더미, 이후 별도 OAuth/OIDC 연동 검토
- 이메일·비밀번호: P1에서는 더미

소셜 계정 배지:

- Kakao: 노랑 + `🐣`
- Naver: 초록 + `🌿`
- Google: 연한 파랑 + `🌈`
- 이메일: 회청색 + `🧩`

### 운영 전 추가 조건

- OAuth callback URL 등록
- 서버 API의 Supabase JWT 검증
- `customer`, `lawyer`, `admin` 서버 권한 분리
- 변호사 역할은 자격 검증 뒤 승격
- 사건 데이터에 사용자 ID와 행 단위 접근정책 적용
- 로그인·공유·삭제 감사 이벤트
- 개인정보 처리방침의 국외 이전·위탁·보관 고지 검토

Supabase Auth 공식 문서상 JWT 인증과 Google·Kakao 소셜 로그인을 지원한다. Auth만 독립적으로 사용할 수 있지만 내부적으로는 관리형 Postgres의 Auth 스키마를 사용한다.

필수 고지:

> 엘파인드 공개 검색·상세 프로필에서 수집해 사실형으로 요약했습니다. 엘파인드 또는 해당 변호사와의 제휴·추천을 뜻하지 않으며, 실제 공개 전 당사자 확인과 최신성 검수가 필요합니다.

자동 정기 스크래핑으로 전환하기 전에는 엘파인드 이용약관·robots 정책·재사용 허용 범위를 별도로 확인한다.

`/admin/dashboard`는 단순 골격이 아니라 다음을 보여준다.

- 기간: 7일·30일·90일
- KPI: 가이드 완료율, AI 결과 완료율, 로그인 전환율, Codex 오류율

- AI 품질: 구조화 성공, 응답시간, 수정 필요, 금지 표현 감지
- 운영 대기열: 프로필 최신성, 콘텐츠 갱신, 삭제 요청, 미응답 상담

- 지표 정의

현재 수치는 목업 집계다. 브라우저의 테스트 이벤트 수만 별도로 표시한다. 실제 운영 연결 시 사건 원문을 통계 화면에 표시하지 않는다.

## 11. 모델 선택

2026-07-21 OpenAI 공식 모델 가이드 기준으로 GPT-5.6 계열을 사용한다.

| 작업 | 권장 모델 | 이유 |
|---|---|---|
| 전체 P1 재기획·안전 경계·복합 리팩터링 | `gpt-5.6-sol`, medium 이상 | 모호하고 고가치인 복합 판단 |
| 명세가 확정된 일반 구현 티켓 | `gpt-5.6-terra`, medium | 일상 구현의 균형 |
| 반복 UI, 데이터 변환, 구조화 출력 | `gpt-5.6-luna`, low~medium | 명확하고 반복 가능한 작업 |
| 제품 런타임 사건 구조화 | `gpt-5.6-luna` 우선 | 고정 JSON Schema와 반복 처리 |

하위 모델에 맡길 티켓에는 다음을 반드시 포함한다.

- 수정 파일과 금지 파일
- 사용자 흐름
- 입력·출력 타입
- 빈 상태·오류 상태·로딩 상태
- 모바일 320px와 키보드 기준
- 법률 금지 표현
- 완료 명령과 수용 기준

먼저 Sol로 대표 예시와 평가 기준을 만들고, Terra 또는 Luna가 같은 품질 목표를 만족하는지 샘플 평가한 뒤 비용·속도를 최적화한다.

## 12. P1 완료 기준

- 홈의 두 CTA가 서로 다른 화면으로 이동한다.
- 여섯 영역마다 후속 질문과 체크리스트가 달라진다.
- 3분 가이드에 설명 3개가 있고 체크리스트만 바로 나오지 않는다.
- AI 질문 전에 답변 주체와 한계를 확인한다.
- 사진은 Codex 이미지 입력으로 전달되고 영상·PDF 제한은 표시된다.
- Codex 결과가 스키마에 맞고 금지 판단을 하지 않는다.
- 결과에서 변호사 프로필과 내 사건으로 이동한다.
- `내 사건`은 미로그인 시 로그인으로 이동한다.
- 소셜 제공자마다 색과 이모티콘이 다르다.
- Supabase 설정이 없어도 더미 로그인으로 전체 흐름이 동작한다.
- 변호사 프로필마다 엘파인드 원문 URL·확인일·수집 고지가 있다.
- 관리자 화면에 퍼널·품질·운영 통계가 있다.
- 320px 모바일에서 가로 잘림 없이 핵심 행동을 사용할 수 있다.
- `npm run build`가 성공한다.
