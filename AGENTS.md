# 억울함해결사 작업 지침

이 폴더는 법률 문제를 겪는 사용자가 사실과 증거를 정리하고, 필요한 다음 행동을 확인하며, 변호사 상담을 준비하도록 돕는 서비스 `억울함해결사`의 프로젝트 루트다.

## 작업 시작 순서

1. `README.md`를 읽는다.
2. `docs/product-plan.md`에서 제품 범위와 안전 원칙을 확인한다.
3. `docs/decisions.md`에서 확정된 결정과 미결정 사항을 확인한다.
4. 구현 결정을 새로 내렸다면 `docs/decisions.md`를 갱신한다.

## 기본 소통 방식

- 사용자와 문서에는 특별한 요청이 없는 한 한국어를 사용한다.
- 결과를 먼저 말하고 필요한 근거와 다음 행동을 명확하게 설명한다.
- 법률·개인정보·보안 관련 결정은 최신 공식 출처를 확인한다.
- 사용자의 기존 파일과 변경사항을 보존하고 무관한 파일을 수정하지 않는다.

## 제품 범위

- 첫 번째 버티컬은 교통사고, 특히 보험사 과실비율 이견과 분심위 이후 단계다.
- 초기 제품은 반응형 웹/PWA를 우선하며 일반 사용자, 변호사, 운영 관리자 화면을 포함한다.
- 무료 공개 영역은 초기 가이드와 템플릿이다.
- 사건 저장, 파일 업로드, AI 정리 등 비용과 개인정보 처리가 발생하는 기능은 로그인 뒤에 둔다.
- AI의 역할은 사실 정리, 누락 자료 확인, 일반 법률정보와 출처 제시다.
- 구체적인 법률자문과 대응 전략은 변호사가 담당한다.

## 법률·AI 안전 원칙

- 승소 가능성, 형량, 합의금, 과실비율을 확정하거나 보장하지 않는다.
- 사용자 진술, 자료에서 추출된 사실, AI 추정을 화면과 데이터에서 구분한다.
- 법령·판례를 제시할 때 출처와 기준일 또는 시행일을 표시한다.
- 근거가 부족하거나 상충하면 답변을 유보하고 추가 자료 또는 변호사 검토를 안내한다.
- 개별 사건 분석 결과를 변호사 검색 순위에 사용하지 않는다.
- 변호사 목록은 객관적 필터와 공개된 중립 정렬 기준을 사용하고 광고 여부를 명확히 표시한다.
- 수임액 비례 수수료, 성공보수 공유, 결과 예측 판매는 구현하지 않는다.

## 개인정보·보안 원칙

- API 키와 비밀값을 프론트엔드 코드에 포함하지 않는다.
- 예제와 테스트에는 실제 이름, 차량번호, 진단서, 사건번호 등 개인정보를 사용하지 않는다.
- 업로드 자료는 최소수집, 접근권한, 보관기간, 삭제, 변호사 공유 동의를 고려해 설계한다.
- 얼굴, 차량번호, 연락처 등 제3자 개인정보의 마스킹 흐름을 고려한다.
- 원본 증거와 AI가 생성한 요약을 별도로 보존하고 출처 연결을 유지한다.

## 구현 원칙

- 가장 작은 완결 사용자 흐름부터 구현한다: `상황 선택 → 무료 체크리스트 → 로그인 유도`.
- 모바일 화면과 접근성을 먼저 확인한다.
- 변호사 소개·목록 등 웹 화면을 새로 만들거나 수정하기 전에는 `docs/references/lfind-lawyer-profile-shin-young-jun/README.md`를 확인한다. 이 자료는 정보 구조와 화면 구성의 참고용이며, 특정 변호사 추천·광고, 사진·문구·개인정보의 복제 근거로 사용하지 않는다.
- 법률 문구와 비상 안내는 가능한 한 규칙 기반 콘텐츠로 관리하고 운영자가 수정할 수 있게 한다.
- AI 응답은 자유 텍스트만 저장하지 말고 검증 가능한 구조화 데이터 형식을 사용한다.
- 새로운 프레임워크나 외부 API를 도입할 때 선택 이유, 비용, 데이터 처리 위치를 기록한다.
- 범위가 불명확할 때는 `docs/product-plan.md`의 P0 우선순위를 따른다.

## 디렉터리 기준

- `apps/web`: 프론트엔드 애플리케이션
- `docs`: 제품, 결정, 조사 문서
- `docs/references`: 외부 화면·서비스의 구조 참고자료. 원문 링크, 확인일, 활용 범위와 주의사항을 함께 기록한다.
- 실제 코드 구조가 확정되면 이 문서와 `README.md`의 구조 설명을 함께 갱신한다.
